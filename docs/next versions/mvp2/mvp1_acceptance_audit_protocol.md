# Протокол аудита и приемки MVP1  
## Проект: браузерный редактор мобильных макетов с импортом/экспортом Draw.io

**Статус:** рабочий документ для Codex AI  
**Назначение:** формализовать, как именно проверять выполнение первого ТЗ (`technical_spec_mobile_mockup_editor_for_codex.md`), чтобы Codex не “принял сам себя” без доказательств.

---

## 0. Что должно быть покрыто этим документом

В рамках аудита MVP1 нужно проверить:

1. Что реализация вообще существует и запускается.
2. Что функциональность соответствует первому ТЗ, а не только общей идее проекта.
3. Что поддерживается именно тот controlled subset Draw.io, который зафиксирован в первом ТЗ и в `drawio_observed_tokens_from_example.json`.
4. Что импорт/экспорт `.drawio` не ломает иерархию, геометрию и базовые стили.
5. Что UI-редактор действительно удобен для базовой работы: board, pan/zoom, screen-as-parent, drag/resize, inspector, layers.
6. Что есть минимально достаточные тесты и документация.
7. Что можно вынести формальное решение: **MVP1 принят / принят с оговорками / не принят**.

## 1. Источники истины

Во время аудита источниками истины считать, в порядке приоритета:

1. `technical_spec_mobile_mockup_editor_for_codex.md` — главное ТЗ MVP1.

3. `one-click-flow-sketch.drawio` — эталонный пример для проверки импорта/экспорта.

6. Реальное поведение приложения в браузере.

Если реализация противоречит ТЗ, правильным считается ТЗ, а не реализация.

## 2. Принципы аудита

### 2.1. Никакой “самоаттестации”
Недостаточно фразы вроде:
- “примерно реализовано”;
- “концептуально есть”;
- “можно быстро доделать”;
- “это подразумевалось”.

Нужно реальное подтверждение:
- код;
- видимое поведение;
- тест;
- экспортированный файл;
- отчет;
- скриншот/описание результата.

### 2.2. Атомарность требований
Первое ТЗ должно быть разложено на **атомарные проверяемые требования**.  
Каждое требование должно отвечать на вопрос:

> можно ли однозначно сказать PASS / PARTIAL / FAIL?

Если нельзя — требование надо декомпозировать.

### 2.3. Полная трассируемость
Каждое атомарное требование должно иметь:
- уникальный ID;
- ссылку на раздел первого ТЗ;
- приоритет;
- тип доказательства;
- статус;
- комментарий.

### 2.4. Разделение “реализовано” и “удобно”
Если функция номинально существует, но пользоваться ей неудобно или она ломает модель, это **не PASS**, а минимум **PARTIAL**.

### 2.5. Приоритетность
Во время приемки использовать уровни критичности:

- **Blocker** — без этого MVP1 нельзя считать принятым.
- **Major** — важная часть MVP1, без неё возможна только условная приемка.
- **Minor** — доп. качество, UX-полировка или неключевая часть.
- **Nice-to-have** — желательно, но не блокирует приемку.

| Evidence Type | code / UI / automated test / manual test / XML output / docs |
| Evidence | конкретные ссылки на файлы / тесты / команды / маршруты |
| Status | PASS / PARTIAL / FAIL / N/A / DEFERRED |
| Severity | Blocker / Major / Minor |
| Notes | что работает / что сломано / что ограничено |

### 3.2. `mvp1_manual_test_protocol.md`
Должен содержать:

- статус.

### 3.3. `mvp1_automated_test_report.md`
Должен содержать:

### 3.4. `mvp1_drawio_roundtrip_report.md`
Должен содержать:
- описание импорта эталонного `.drawio`;

- список потерь и ограничений.

### 3.5. `mvp1_gap_list.md`
Список всех незакрытых требований с группировкой:
- Blockers
- Major gaps

- Known limitations

Итоговое решение:
- принят / не принят / принят условно;
- summary по числу PASS/PARTIAL/FAIL;
- список блокеров;
- список major gaps;
- что обязательно закрыть в MVP2.

## 4. Алгоритм разложения первого ТЗ на атомарные требования

### Шаг 1. Извлечь все требования из первого ТЗ
Нужно пройти по разделам первого ТЗ и выписать требования минимум из разделов:

- раздел 5 — UX и поведение редактора;
- раздел 6 и 7 — каталог элементов;
- раздел 8 — визуальный стандарт;
- раздел 9 и 10 — модель данных и вложенность;
- раздел 11 и 12 — import/export и XML mapping;
- раздел 14 — алгоритмы поведения;
- раздел 17, 18, 19 — persistence/history/validation;
- раздел 21 — тестирование;
- раздел 22 — acceptance criteria.

### Шаг 2. Раздробить крупные пункты
Пример:  
“Есть import/export `.drawio`” — это слишком крупно.  
Его надо разбить минимум на:

- импорт одного `.drawio` файла работает;
- XML корректно парсится;
- screen распознаётся;
- text распознаётся;
- edges распознаются;
- style keys из поддерживаемого subset маппятся;
- экспорт обратно создаёт валидный `.drawio`;
- поддерживаемые элементы не теряются бесшумно;
- unsupported elements получают fallback/warning.

### Шаг 3. Назначить ID
Рекомендуемый формат:

```text
V1-<section>-<group>-<number>
```

Примеры:
- `V1-05-BOARD-001`
- `V1-05-SCREEN-003`
- `V1-11-IMPORT-007`
- `V1-22-AC-004`

### Шаг 4. Назначить приоритет
Рекомендуемое правило:

- Всё из раздела **22 Acceptance criteria / Обязательно** = минимум **Blocker** или **Major**.
- Всё, без чего нельзя пользоваться редактором как продуктом, = **Blocker**.
- Всё, без чего редактор работает, но заметно слабее или неудобнее, = **Major**.
- Полировка и частично необязательные части = **Minor**.

### Шаг 5. Назначить тип доказательства
У каждого требования должен быть основной тип доказательства:

- `UI behavior`
- `Automated test`
- `Manual test`
- `Code inspection`
- `Exported XML`
- `Documentation`
- `Performance run`

### Шаг 6. Назначить критерий PASS / PARTIAL / FAIL
Нужно зафиксировать заранее:
- что считается полным выполнением;
- что считается частичным выполнением;
- что считается провалом.

---

## 5. Статусы проверки

Использовать только эти статусы:

- **PASS** — требование выполнено полностью и доказано.
- **PARTIAL** — реализовано частично, с ограничениями, UX-проблемой или нестабильностью.
- **FAIL** — требование не выполнено или сломано.
- **N/A** — не применимо, только если это действительно обосновано.
- **DEFERRED** — сознательно отложено; допускается только для Minor/Nice-to-have, но не для Blocker.

Запрещены расплывчатые статусы:
- “почти готово”;
- “концептуально сделано”;
- “на 80%”.

---

## 6. Количественные правила приемки

### 6.1. MVP1 считается **не принятым**, если:
- есть хотя бы **1 Blocker = FAIL**;
- есть хотя бы **1 Blocker = PARTIAL** без формально документированного обходного пути;
- приложение не запускается;
- отсутствует import или export `.drawio`;
- отсутствует screen containment;
- редактор ломает parent-child при базовых операциях;
- round-trip эталонного файла приводит к потере поддерживаемых ключевых сущностей без предупреждения.

### 6.2. MVP1 может считаться **принятым условно**, если:
- все Blocker = PASS;
- Major PASS-rate не ниже **90%**;
- остаточные проблемы — только Minor либо явно задокументированные Major без потери данных;
- есть четкий список gap-ов для MVP2.

### 6.3. MVP1 считается **принятым полноценно**, если:
- все Blocker = PASS;
- Major PASS-rate не ниже **95%**;
- нет data-loss на поддерживаемом subset;
- тесты и документация присутствуют;
- реализация соответствует первому ТЗ по сути, а не только на уровне интерфейсного “демо”.

## 7. Обязательные группы проверки

Ниже — минимальный список категорий, который нельзя пропускать.

---

## 7.1. Проверка запуска и каркаса приложения

Проверить:
1. Проект устанавливается.
2. Проект запускается локально.
3. Открывается редактор в браузере.
4. Нет фатальных ошибок при старте.
5. Есть базовый layout: top bar, board, sidebar(ы) — если это требовалось ТЗ.
6. Приложение не пустое и не заглушка.

**Тип доказательства:** runtime + скриншот/описание + код.

---

## 7.2. Проверка board / viewport

Проверить:
1. Есть бесконечная или достаточно большая рабочая доска.
2. Работает pan:
   - middle mouse, либо
   - space + drag, либо
   - другой документированный механизм.
3. Работает zoom.
4. Есть fit board / fit selection / fit screen — если заявлено.
5. Есть grid.
6. Есть snap to grid.
7. Есть smart guides — если заявлено.

**PASS:** board пригодна для реальной работы.  
**PARTIAL:** nominally работает, но поведение неудобное/ломкое.  
**FAIL:** нет нормальной навигации или холст фактически непригоден.

---

## 7.3. Проверка screen как родительского контейнера

Проверить:
1. Можно создать screen.
2. Можно менять width/height screen.
3. У screen есть визуальная рамка.
4. В screen можно добавлять элементы.
5. Элемент получает parent = screen или container внутри screen.
6. Child coordinates хранятся относительно parent.
7. При перемещении screen дети остаются на screen.
8. Child нельзя случайно утащить за пределы screen.
9. Reparent работает корректно либо явно ограничен и документирован.

Это один из главных блокеров.

---

## 7.4. Проверка палитры и добавления элементов

Минимально проверить элементы из первого ТЗ:

- Text
- Box / Section
- Field
- Checkbox
- Button
- SegmentedControl
- Badge
- Banner
- Arrow

Если в MVP1 какие-то элементы объявлены поддержанными, но не создаются через UI, это не PASS.

---

## 7.5. Проверка composite-компонентов

Composite-компоненты должны не быть просто “одним прямоугольником”, если в ТЗ они описаны как составные.

Проверить:
- Field = container + label + value;
- CheckboxRow = checkbox + text;
- SegmentedControl = container + items + active state;
- Banner = container + title/description/action if applicable;
- ProductRow / InfoRow — если входят в поддержанный набор.

Нужно проверить:
- корректную внутреннюю структуру;
- редактируемость;
- экспорт/импорт без разрушения структуры.

---

## 7.6. Проверка текста

Проверить:
1. Текст можно создавать.
2. Текст можно редактировать.
3. Поддерживаются базовые text props:
   - text
   - align
   - font size
   - font weight
   - color
4. HTML-based value корректно обрабатывается, если это обещано.
5. Стили не теряются при export/import.

---

## 7.7. Проверка слоев и дерева

Проверить:
1. Есть layers/tree panel.
2. Видна иерархия:
   - board / lane / screen / children.
3. Можно выделить элемент через дерево.
4. Можно понять, кто чей parent.
5. Можно безопасно менять parent, если функция заявлена.
6. Нет “невидимых сирот” без parent.

---

## 7.8. Проверка inspector

Проверить:
1. Есть inspector выбранного элемента.
2. Можно менять:
   - x / y
   - width / height
   - fillColor
   - strokeColor
   - borderRadius
   - text / label / value
3. Изменения применяются предсказуемо.
4. Изменения не ломают модель.

---

## 7.9. Проверка undo/redo и clipboard

Проверить:
1. Undo работает.
2. Redo работает.
3. Drag не засоряет историю тысячами шагов.
4. Copy/paste или duplicate работают, если заявлены.
5. После undo/redo model consistency не ломается.

---

## 7.10. Проверка persistence

Проверить:
1. Save JSON работает.
2. Load JSON работает.
3. Local autosave, если заявлен, реально работает.
4. После перезагрузки данные можно восстановить.
5. Документ не повреждается.

---

## 7.11. Проверка validation

Проверить:
1. Есть validation engine.
2. Ловятся:
   - duplicate IDs
   - invalid nesting
   - missing required props
   - invalid edge endpoints
   - geometry <= 0
   - orphan children
3. Ошибки различаются как:
   - fatal
   - warning
   - info
4. Export запрещается при fatal-errors — если это предусмотрено.

---

## 7.12. Проверка import `.drawio`

Проверить:
1. Файл `one-click-flow-sketch.drawio` импортируется.
2. Поддерживаемые элементы распознаются.
3. Screen-фреймы распознаются корректно.
4. Swimlane/FlowLane распознаются корректно.
5. Text node распознаются.
6. Edge/connector распознаются.
7. Cloud note корректно обрабатывается либо документированно деградирует.
8. Unsupported styles не теряются молча.
9. Есть warnings/report по неподдержанному.

---

## 7.13. Проверка export `.drawio`

Проверить:
1. Из internal JSON создаётся валидный `.drawio`.
2. Экспортированный файл можно открыть в diagrams.net/draw.io.
3. Поддерживаемые элементы сериализуются в допустимый mxGraph XML.
4. Сохраняются:
   - geometry
   - parent-child
   - основные styles
   - text/value
5. У composite-компонентов экспортированная структура соответствует заявленным mapping rules.

---

## 7.14. Проверка round-trip

Прогнать сценарий:

```text
source drawio -> import -> internal JSON -> export -> drawio
```

Проверить минимум:

- screen count не потерян;
- lane count не потерян;
- edge count не потерян на поддерживаемом subset;
- parent-child логика сохранена;
- основные style keys из поддержанного списка сохранены;
- text не исчез;
- geometry не “улетела”.

### Рекомендуемые метрики
В `mvp1_drawio_roundtrip_report.md` зафиксировать:

- imported_supported_nodes
- exported_supported_nodes
- supported_node_preservation_rate
- supported_style_key_preservation_rate
- unsupported_nodes_reported_count
- fatal_roundtrip_breakages_count

### Минимальные целевые значения
- `supported_node_preservation_rate >= 0.98`
- `supported_style_key_preservation_rate >= 0.95`
- `fatal_roundtrip_breakages_count = 0`

Если команда сознательно делает нормализацию, это допустимо только если:
- документировано;
- нет семантической потери;
- нет silent data loss.

---

## 7.15. Проверка design tokens и визуального стандарта

Проверить, используются ли хотя бы базовые токены из `drawio_observed_tokens_from_example.json`:

### Цвета
- `#1c2a30` — screen stroke
- `#ffffff` — surface
- `#f7fafb` — muted surface
- `#eef5f4` — header tint
- `#dcefeb` — interactive fill
- `#0f766e` — interactive stroke / success text
- `#d5e8d4` — success fill
- `#82b366` — success stroke
- `#ffe6cc` / `#d79b00` — warning
- `#fffaf0` / `#c49102` — note cloud

### Скругления
- `arcSize 6, 8, 9, 10, 12, 14, 16`

### Типографика
- 9px label
- 10px section title
- 11px body / CTA
- 14px display title

Если реализация вообще игнорирует эти токены и делает случайный стиль, это как минимум **PARTIAL**.

## 8. Алгоритм ручной приемки

Ниже — практическая последовательность, в которой проводить проверку.

### Этап A. Статический аудит репозитория
1. Найти README, package.json, tsconfig, структуру папок.
2. Проверить наличие docs и tests.
3. Проверить наличие модулей:
   - editor
   - import/export
   - model
   - inspector
   - layers
4. Проверить strict TypeScript и отсутствие очевидного хаоса.

### Этап B. Smoke run
1. Установить зависимости.
2. Запустить проект.
3. Открыть приложение.
4. Проверить отсутствие фатальных ошибок.

### Этап C. Базовая функциональная проверка
1. Создать новый документ.
2. Создать screen.
3. Добавить text.
4. Добавить field.
5. Добавить checkbox.
6. Добавить button.
7. Подвигать элементы.
8. Попробовать унести child за пределы screen.
9. Изменить свойства через inspector.
10. Проверить tree/layers.

### Этап D. History / persistence
1. Undo/redo.
2. Save/load JSON.
3. Перезагрузка.

### Этап E. Import/export
1. Импортировать `one-click-flow-sketch.drawio`.
2. Проверить распознавание основных элементов.
3. Экспортировать обратно.
4. Проверить XML и открыть его.

### Этап F. Round-trip / fidelity
1. Сравнить counts.
2. Сравнить style keys.
3. Сравнить хотя бы 3 representative screens.
4. Сравнить 1 banner, 1 checkbox-row, 1 button, 1 cloud note.

### Этап G. Acceptance decision
1. Заполнить traceability matrix.
2. Сформировать gap list.
3. Сформировать final acceptance report.

## 10. Правила оценки PARTIAL

Статус **PARTIAL** ставить в этих случаях:

- функция работает только частично;
- функция работает, но ломает модель;
- функция работает только для части кейсов;
- функция есть, но не через UI, а только через внутренний API;
- функция работает, но UX настолько плох, что практическое использование затруднено;
- функция не соответствует ТЗ по степени завершенности.

Нельзя завышать PARTIAL до PASS.

---

## 11. Что считать Blocker-ами

Минимальный набор Blocker-ов для MVP1:

1. Приложение не запускается.
2. Нет board с pan/zoom.
3. Нельзя создать screen.
4. Не работает containment внутри screen.
5. Нельзя редактировать основные свойства.
6. Нет import `.drawio`.
7. Нет export `.drawio`.
8. Parent-child ломается при drag/reparent.
9. Undo/redo отсутствует или ломает документ.
10. Нет validation хотя бы на базовом уровне.
11. Поддерживаемые сущности теряются при round-trip без предупреждения.

---

## 12. Автоматизированные проверки, которые обязательно должны существовать

В проекте должны быть тесты минимум на:

- parseStyle / serializeStyle
- parseGeometry / serializeGeometry
- import/export Screen
- import/export Field
- import/export CheckboxRow
- import/export Button
- import/export Edge
- bounds clamping
- reparent coordinate conversion
- deterministic IDs
- basic editor smoke
- import/export round-trip of representative subset

Если каких-то тестов нет, это не обязательно FAIL, но это должно понизить итоговую оценку до PARTIAL там, где доказательств недостаточно.

---

## 13. Что делать с несовпадениями между первым ТЗ и кодом

Если код уже написан, но:
- структура отличается,
- naming отличается,
- UI отличается,
- часть функций сделана другим способом,

то это допустимо только если:
1. требование первого ТЗ реально закрыто;
2. поведение не хуже;
3. модель не ломается;
4. ограничения документированы.

Если это просто “сделали по-другому и хуже” — это FAIL/PARTIAL.

## 14. Решение по итогам аудита

В `mvp1_final_acceptance_report.md` должен быть один из трёх итогов:

### 14.1. `ACCEPTED`
Использовать только если:
- все Blocker = PASS;
- Major >= 95% PASS;
- нет data-loss на поддерживаемом subset;
- есть документы и тесты.

### 14.2. `ACCEPTED_WITH_GAPS`
Использовать, если:
- все Blocker = PASS;
- есть ограниченное число Major gaps;
- есть четкий план закрытия в MVP2.

### 14.3. `REJECTED`
Использовать, если:
- есть хотя бы один незакрытый Blocker;
- либо product exists only as demo, а не как рабочий редактор.
