# Hard UX UI Review

This document was generated with the assistance of Codex AI and prompted by Ustselemov.

## Review scope

- repository shell UI
- left library and right inspector/layers workflow
- canvas prominence and information hierarchy
- architecture signals that directly affect UI maintainability

## Critical findings

### P1. Canvas was not the dominant surface

The previous shell spent too much area on repeated chrome, especially the top command band and oversized side panels. The editor had capable functionality, but the first impression was "toolbox around a canvas" instead of "canvas with supporting tools".

### P1. The library scanning pattern was too slow

The old two-column library grid produced many tiny cards with weak scan rhythm. Users had to decode badge, preview, title, and description in a cramped layout, which slowed insertion and made the left rail feel denser than it really was.

### P1. Status information was duplicated and noisy

Screens, validation, selection, debug state, and autosave were split across multiple areas with overlapping meaning. The result was unnecessary reading load for routine operation.

### P2. The shell hierarchy was visually flat

Toolbar groups, side panels, and inspector surfaces shared too much of the same visual weight. That reduced the sense of primary vs secondary workspace and made the editor feel heavier than the underlying interaction model.

### P2. Documentation drift is already visible

Some documentation describes UI gaps that the current codebase has already closed, for example historical notes about the minimap. That drift increases review cost because the repo has many docs and no single "latest audit" anchor.

## Architecture findings

### P1. UI orchestration is concentrated in large files

The shell still relies on several high-mass files:

- `src/app/App.tsx`
- `src/app/app.css`
- `src/core/store/editorStore.ts`
- `src/features/editor/components/BoardView.tsx`

This is not a correctness failure today, but it is a clear maintainability risk. Shell redesign, command-surface changes, and board interaction changes are coupled more tightly than they should be.

### P2. Global CSS is carrying feature-local responsibility

`src/app/app.css` owns styling for the shell, palette, inspector, layers, minimap, quick insert, and board overlays. That makes localized redesign work expensive and increases the chance of unintended regressions.

## Implemented in this pass

- compressed the shell hierarchy so the canvas reads as the primary surface
- moved high-signal metadata into compact header metrics
- reduced sidebar widths to return more area to the board
- reworked library cards into a denser single-column scan pattern
- clarified palette insertion context with explicit "Insert into ..." copy
- simplified the status strip into compact operational chips
- reduced minimap footprint
- added project favicon assets to remove the startup missing-icon error path

## Implemented in the follow-up pass

- removed the bottom status strip completely instead of compressing it
- introduced a dedicated mobile workspace mode with a large canvas and a single switchable panel area
- made library previews more visual and less text-led
- added explicit parent detachment via "Move to board" in the inspector
- reduced mobile shell control density by hiding secondary fit controls on narrow widths

## Next refactor plan

1. Extract shell chrome from `src/app/App.tsx` into dedicated app-shell components.
2. Split `src/app/app.css` into feature-owned styles or CSS modules by shell area.
3. Break `editorStore.ts` into document mutations, history, persistence, and import/export slices.
4. Separate `BoardView.tsx` into interaction controllers, overlay rendering, and inline editing.
5. Add a small visual regression baseline for the shell at desktop and narrow widths.

## Recommended immediate follow-ups

- add collapsible or resizable sidebars so users can bias toward canvas when needed
- promote a focused command model for file actions vs canvas actions instead of giving both equal visual weight
- make the inspector selection state more expressive, especially for empty and multi-select cases
- unify documentation around a current-state entrypoint so old MVP notes stop competing with active behavior
